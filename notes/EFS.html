# 架构设计

# 名字节点

名字节点主要存放元数据相关数据, 元数据主要记录两类关系, 分别为第一关系和第二关系,

第一关系记录文件与数据块列表的对应关系,

第二关系记录数据块与节点信息的对应关系

# 数据节点

数据节点是存放具体数据的设备, 数据节点中每个数据块中会保存数据和校验和, 这样数据节点可以自己进行校验, 检查数据是否损坏,

# 系统设计

# 读流程

客户端从名字节点获取存放一个数据块的数据节点,

客户端从数据节点读取数据

如果数据节点不在线, 读取副本

客户端获取下一个数据块的数据节点

# 写流程

客户端向名字节点申请一个数据块,

客户端向名字节点分配的数据节点写入数据

如果数据节点无法连接

客户端abandonBlock, 重新向名字节点申请数据块

客户端向名字节点分配的数据节点写入数据

客户端向名字节点申请一个数据块,

# 删除

客户端向名字节点删除文件

名字节点将文件相关数据块标记为删除

数据节点心跳连接名字节点,

名字节点下发删除

# 数据恢复

1 名字节点检测到副本不够 通知一个拥有该数据块的数据节点, 复制到其他数据节点 2 sdk, 数据节点全量汇报检测数据出错 通知一个拥有该数据块的数据节点, 复制到其他数据节点

# 数据节点挂掉

如果数据节点在sdk写入后, 数据节点向名字节点汇报前挂掉, 名字节点感知到错误

假如数据块存在于数据节点1, 数据节点2, 数据节点3

数据节点1挂掉,

则数据节点2, 数据节点3汇报时, 名字节点会指定数据节点2复制到一个正常节点

通知一个拥有该数据块的数据节点, 复制到其他数据节点

如果数据节点在sdk写入时挂掉, sdk感知到错误:

会选择一个主数据节点, 然后让主数据节点对其他数据节点进行数据复制

通知一个拥有该数据块的数据节点, 复制到其他数据节点

之后数据节点会返回sdk一个恢复成功数据块的位置, 和数据节点列表

sdk可以在这个位置, 继续写入

# sdk挂掉

参考租约恢复

# 名字节点挂掉,

sdk在写入时, 名字节点挂掉

名字节点只能在给sdk分配完数据节点后挂掉, 如果之前挂, 则sdk无法找到写的数据节点

此时, sdk把该数据块写完后, 无法关闭文件, 需人工介入,

数据节点这里无所谓, 只是暂时块汇报无法响应

# 高可用, 备机做fsimage

备机从主机拉取editlog和image合成新的image,
再将新的image推送给主机

# 注册

在注册时, 上报本节点的数据块信息, 名字节点根据数据块信息, 构建{数据块:节点信息}

# 心跳

每次心跳, 名字节点向数据节点发送无效节点信息列表, 数据节点进行删除,

# 并发, 租约

# 租约恢复

软超时, 是sdk每隔2s进行续约

硬超时, 是如果sdk断开, 且没有其他sdk抢夺这个租约造成的超时,

一般是1h

如果sdk刚打开文件就挂了, 还没写数据:

直接关闭文件

abandonBlock调用场景:

当sdk请求一个数据块后, 名字节点返回一个数据节点,

当sdk无法连接这个数据节点后, 就会调用abandonBlock,

让名字节点重新分配一个数据节点

如果sdk在调用abandonBlock后挂掉:

主数据节点是倒数第二个数据块的数据节点(倒数第一个被abandon了)

指定一个主数据节点, 然后主数据节点与其他数据节点计算出一个最短公共长度,

以此最短公共长度作为有效数据,

主数据节点向名字节点申请一个新的版本号, 并同步给其他数据节点, 将这个版本号赋给数据块
